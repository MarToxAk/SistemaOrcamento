Substituir esse bloco por logica nova com tres caminhos de resolucao:

**Caminho A — clienteAthosId explicito (per TOMAD-01, TOMAD-02):**
Quando `input?.clienteAthosId` esta presente e e um numero > 0:
1. Chamar `this.athosService.buscarClientePorId(input.clienteAthosId)`
2. Se retornar null: lancar `BadRequestException("Cliente Athos não encontrado. Verifique o clienteAthosId informado.")`
3. Preencher tomadorCnpj, tomadorCpf, tomadorNome e tomadorEndereco com os dados retornados
4. Aplicar mesmas regras de documento e endereco do fluxo manual:
   - Se `info.type === "juridico"` e documento tem 14 chars → cnpj
   - Se `info.type === "fisico"` e documento tem 11 chars → cpf
   - Preservar `tomadorEndereco` do `input` se o usuario informou explicitamente (buildTomadorEnderecoFromInput)

**Caminho B — documento manual informado (fluxo ja existente):**
Manter comportamento atual: `documentoManualInformado = Boolean(tomadorCnpj || tomadorCpf)` → buscar apenas endereco via `buscarTomador` se nao informado no input.

**Caminho C — nenhum clienteAthosId nem documento manual (fluxo ja existente):**
Manter chamada a `buscarTomador(quote)` como antes.

**Mudanca 3 — validacao pos-resolucao (per TOMAD-04):**

// TOMAD-04: validar documento e endereco minimo obrigatorio

}
