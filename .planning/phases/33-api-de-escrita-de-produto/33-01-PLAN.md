# Plan 33-01: Infraestrutura Base e Endpoint POST (Criar Produto)

## Objective
Criar a infraestrutura de escrita no Athos (`AthosProdutoService` com pool próprio, DTOs `CreateProdutoDto`/`UpdateProdutoDto`, fail-fast da env var `ATHOS_SISTEMA_USUARIO_ID`) e implementar o método `criarProduto` com INSERT serial + RETURNING, pre-validação de FK e log estruturado, cobrindo CPROD-01..04 e SPROD-01.

## must_haves
<truths>
- `AthosProdutoService` existe com pool lazy-init próprio (`getPool()` + `getDbConfig()`), replicando o padrão de `athos.service.ts` L500-528.
- `CreateProdutoDto` contém os 27 campos curados de D-04 (obrigatório: `descricaoproduto`); `descontomaximo` validado com `@Min(0) @Max(100)`.
- `UpdateProdutoDto extends PartialType(CreateProdutoDto)` de `@nestjs/swagger` (já criado nesta wave para uso na Wave 2).
- `criarProduto(dto)` emite um único INSERT na tabela `produto` (sem `idproduto` no payload), com `datacadastro = NOW()`, `idusuariocadastro = $env`, `idusuarioalteracao = $env`, e retorna `{ idproduto }` via `RETURNING idproduto`.
- FK informada inválida (`iddepartamento`/`idgrupo`/`idmarca`) gera `UnprocessableEntityException` (422) antes do INSERT, via pre-query `SELECT 1 FROM <tabela> WHERE <coluna> = $1 LIMIT 1` (per D-08).
- Nenhuma query emitida por `criarProduto` contém `LOCK TABLE`, `DISABLE TRIGGER`, `DELETE`, ou nome de tabela diferente de `produto` / `produto_departamento` / `produto_grupo` / `produto_marca` (SPROD-01, CPROD-03, DPROD-03).
- `ATHOS_SISTEMA_USUARIO_ID` está em `REQUIRED_ENV_VARS` de `app.module.ts` (fail-fast, D-01, D-02).
- `AthosProdutoService` registrado em `providers` e `exports` de `athos.module.ts` (D-03).
- `client.release()` chamado em `finally` em todos os caminhos (sucesso e erro).
- Testes co-locados cobrem: INSERT retorna idproduto do RETURNING; FK inválida lança 422; pool.connect falha lança erro; nenhuma query contém DELETE/DISABLE TRIGGER.
</truths>

<threat_model>
**Threats:**
- SQL injection via campos do DTO (Tampering): mitigado por queries parametrizadas `$n` — nenhuma concatenação de valor de usuário em string SQL.
- Write em tabela não autorizada (Tampering/Elevation): mitigado — `AthosProdutoService` só emite SQL com nomes de tabela literais hardcoded (`produto`, `produto_departamento`, `produto_grupo`, `produto_marca`); nenhum nome de tabela vem do input HTTP.
- `descontomaximo` fora de range (Tampering): mitigado por `@Min(0) @Max(100)` no DTO (400) + CHECK constraint `ckc_descontomaximo_produto` no banco (dupla proteção).
- `ATHOS_SISTEMA_USUARIO_ID` ausente em runtime (DoS por má configuração): mitigado por fail-fast em `REQUIRED_ENV_VARS` (D-02) — sistema não sobe sem o valor.
- `idusuariocadastro` FK inválida por env mal configurada (código 23503): capturado no catch e, quando a constraint violada é a de usuário, convertido em `InternalServerErrorException` com mensagem orientada à configuração. FK de input do operador (`idfornecedor`/`idunidade`) é convertida em `UnprocessableEntityException` (422), nunca mascarada como erro de configuração.
</threat_model>

Importar de `@nestjs/common`: `BadRequestException`, `Injectable`, `InternalServerErrorException`, `Logger`, `NotFoundException`, `UnprocessableEntityException`. Importar `Pool`, `PoolClient` de `pg`, e `CreateProdutoDto` / `UpdateProdutoDto` dos DTOs.

Membros privados: `logger = new Logger(AthosProdutoService.name)`, `_pool: Pool | null = null`. Replicar `getPool()` e `getDbConfig()` de `athos.service.ts` L500-528 — pool próprio (não injetado), `max: 5, idleTimeoutMillis: 30000, connectionTimeoutMillis: 5000`, `_pool.on("error", ...)`. `getDbConfig()` lê `ATHOS_PG_HOST/DB/USER/PASS/PORT` e lança `InternalServerErrorException` se faltar host/database/user/password.

Helper privado `validarFkExiste(client, tabela, coluna, id, nomeEntidade)`: executa `SELECT 1 FROM "<tabela>" WHERE "<coluna>" = $1 LIMIT 1` com `[id]`; se `rows.length === 0` lança `UnprocessableEntityException` com mensagem `"<nomeEntidade> com id <id> nao encontrado no Athos"`. Os nomes de tabela/coluna vêm SEMPRE de literais hardcoded no service — nunca do input HTTP (per threat model).

Método `async criarProduto(dto: CreateProdutoDto): Promise<{ idproduto: number }>`:
1. `const sistemaUsuarioId = Number(process.env.ATHOS_SISTEMA_USUARIO_ID)`.
2. `pool.connect()` → `client`, abrir `try`.
3. Pre-validação FK: se `dto.iddepartamento !== undefined` → `validarFkExiste(client, "produto_departamento", "iddepartamento", dto.iddepartamento, "Departamento")`; idem `idgrupo`→`produto_grupo`/`"Grupo"`; `idmarca`→`produto_marca`/`"Marca"`.
4. INSERT estático na tabela `produto`: colunas fixas `descricaoproduto, datacadastro, idusuariocadastro, idusuarioalteracao` + todos os campos opcionais do DTO que estejam definidos. NÃO incluir `idproduto` (serial — DEFAULT). `datacadastro` recebe `NOW()` literal no SQL (não parâmetro — Pitfall 3). `idusuariocadastro` e `idusuarioalteracao` ambos recebem `$2` (= `sistemaUsuarioId`). Construir lista de colunas/valores dinamicamente a partir das chaves de `dto` com `value !== undefined`, sempre com placeholders `$n` parametrizados. Terminar com `RETURNING idproduto`. NÃO usar `LOCK TABLE`, `BEGIN`, `COMMIT` (idproduto é serial — Anti-pattern do RESEARCH.md).
5. `idproduto = result.rows[0].idproduto`. `this.logger.log(\`criarProduto descricao="${dto.descricaoproduto}" idproduto=${idproduto} idusuario=${sistemaUsuarioId}\`)`. Retornar `{ idproduto }`.
6. `catch (err)`: `this.logger.error`. Re-lançar se `err instanceof BadRequestException || UnprocessableEntityException || NotFoundException`. Se `(err as any).code === "23503"` (foreign_key_violation), distinguir a FK violada inspecionando `(err as any).constraint` — o driver `pg` preenche o nome da constraint violada no objeto de erro (campo `constraint`). Lógica de disposição:
   - Se `constraint` corresponder à FK de usuário do sistema (`idusuariocadastro`/`idusuarioalteracao` — constraints `fk_produto_relations_funciona` e `fk_produto_funcionar_funciona`), lançar `InternalServerErrorException("ATHOS_SISTEMA_USUARIO_ID invalido — usuario nao encontrado em funcionario_usuario")` — é problema de configuração do sistema, não de input do operador.
   - Para qualquer outra constraint FK (input do operador: `idfornecedor`, `idunidade`, etc.), lançar `UnprocessableEntityException(\`FK invalida: campo ${(err as any).constraint} nao encontrado\`)` (HTTP 422, Q3 do RESEARCH.md) — mensagem clara indicando o campo FK violado.
   - Se `constraint` for `undefined`/ausente em um erro 23503, tratar como FK de input do operador (422), nunca como erro de configuração — não mascarar input inválido como 500.
   Para qualquer `err` que não seja 23503 nem uma exceção re-lançada acima, lançar `InternalServerErrorException("Erro ao criar produto no Athos")`.
7. `finally { client.release(); }`.

Editar `athos.module.ts`: importar `AthosProdutoService`; adicioná-lo a `providers` e `exports` (D-03). Não remover `AthosService` nem `AthosListenerService`.
</action>
