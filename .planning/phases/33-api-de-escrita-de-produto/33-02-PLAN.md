# Plan 33-02: Endpoint PATCH /:idproduto (Editar Produto)

## Objective
Implementar o método `editarProduto(idproduto, dto)` no `AthosProdutoService` com partial update dinâmico, verificação de existência, pre-validação de FK, injeção obrigatória de `idusuarioalteracao` e log estruturado, cobrindo EPROD-01..04.

## must_haves
<truths>
- `editarProduto(idproduto, dto)` verifica existência do produto com `SELECT 1 FROM produto WHERE idproduto = $1 LIMIT 1` e lança `NotFoundException` (404) se não existir.
- O UPDATE constrói o SET dinamicamente apenas com campos `value !== undefined` do `UpdateProdutoDto`, todos parametrizados `$n` (EPROD-01: preços; EPROD-02: cadastro).
- `idusuarioalteracao = $n` (= `ATHOS_SISTEMA_USUARIO_ID`) está SEMPRE presente no SET, independente do conteúdo do DTO (Pitfall 2 do RESEARCH.md — evita violação FK e satisfaz parte de EPROD-03).
- `dataultimaalteracao` / `horaultimaalteracao` NUNCA são enviados no UPDATE — a trigger `tg_alterarproduto` os gerencia (EPROD-03).
- O UPDATE tem alvo exclusivamente a tabela `produto`; nenhuma outra tabela do Athos recebe write (D-06, EPROD-04).
- FK informada inválida no PATCH (`iddepartamento`/`idgrupo`/`idmarca`) gera 422 antes do UPDATE.
- `client.release()` chamado em `finally` em todos os caminhos.
- Testes cobrem: edição de preços; edição de cadastro; idusuarioalteracao sempre no SET; dataultimaalteracao nunca enviado; produto inexistente → 404; nenhum write fora de `produto`.
</truths>

<threat_model>
**Threats:**
- SQL injection via nomes de coluna do partial update (Tampering): mitigado — as chaves iteradas vêm de `Object.entries(dto)` onde `dto` é uma instância de `UpdateProdutoDto` validada pelo `ValidationPipe` (objeto TypeScript controlado); propriedades extras são removidas pelo `whitelist` do pipe. Nomes de coluna entre aspas duplas; valores sempre parametrizados `$n`.
- Update de tabela não autorizada (Tampering/Elevation): mitigado — string `UPDATE produto SET ...` com nome de tabela literal hardcoded.
- Edição sobrescrevendo `idproduto`/`datacadastro`/`idusuariocadastro` (Tampering): mitigado — esses campos não existem em `UpdateProdutoDto` (imutáveis por D-05); nunca entram no SET.
- `idusuarioalteracao` NULL causando violação FK na trigger (Integrity): mitigado — sempre incluído no SET via env var.
**Residual risks:** Baixo — comportamento exato da função PL/pgSQL `alterarproduto()` (BEFORE UPDATE) não é totalmente conhecido (A2 do RESEARCH.md); verificado por teste de integração manual na verificação de fase.
</threat_model>

<action>
Adicionar método `async editarProduto(idproduto: number, dto: UpdateProdutoDto): Promise<{ idproduto: number }>` ao `AthosProdutoService` (criado na Plan 01). Reutilizar `getPool()` e `validarFkExiste()` existentes.

1. `const sistemaUsuarioId = Number(process.env.ATHOS_SISTEMA_USUARIO_ID)`.
2. `pool.connect()` → `client`, abrir `try`.
3. Verificar existência: `SELECT 1 FROM produto WHERE idproduto = $1 LIMIT 1` com `[idproduto]`; se `rows.length === 0` lançar `NotFoundException(\`Produto ${idproduto} nao encontrado\`)`.
4. Pre-validação FK condicional (mesma do criarProduto): se `dto.iddepartamento !== undefined` → `validarFkExiste(client, "produto_departamento", "iddepartamento", dto.iddepartamento, "Departamento")`; idem `idgrupo`/`idmarca`.
5. Construir SET dinâmico: `const setClauses: string[] = []; const params: unknown[] = []; let paramIndex = 1;`. Primeiro empurrar SEMPRE `idusuarioalteracao = $${paramIndex++}` com `params.push(sistemaUsuarioId)` (Pitfall 2). Depois `for (const [key, value] of Object.entries(dto))` com `if (value !== undefined)`: `setClauses.push(\`"${key}" = $${paramIndex++}\`)` e `params.push(value)`. NÃO incluir `dataultimaalteracao`/`horaultimaalteracao` (a trigger cuida — EPROD-03); como não estão no DTO, isso é garantido naturalmente.
6. `params.push(idproduto)` (último, para o WHERE). Executar `UPDATE produto SET ${setClauses.join(", ")} WHERE idproduto = $${paramIndex}` com `params`. Tabela `produto` literal (EPROD-04).
7. `this.logger.log(\`editarProduto idproduto=${idproduto} campos=[${Object.keys(dto).join(",")}] idusuario=${sistemaUsuarioId}\`)`. Retornar `{ idproduto }`.
8. `catch (err)`: re-lançar se `instanceof BadRequestException || UnprocessableEntityException || NotFoundException`; senão `InternalServerErrorException("Erro ao editar produto no Athos")`.
9. `finally { client.release(); }`.
</action>

- A string do UPDATE usa `UPDATE produto SET` (tabela literal) e não referencia `dataultimaalteracao`/`horaultimaalteracao`.

- `cd apps/backend && npx tsc --noEmit` exits 0.
