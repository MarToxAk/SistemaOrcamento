# Plan 33-03: Endpoint PATCH /:idproduto/status (Desativar/Reativar)

## Objective
Implementar o método `alterarStatusProduto(idproduto, ativo)` no `AthosProdutoService` que faz toggle de `statusproduto`/`vendeproduto` via UPDATE (nunca DELETE físico), com verificação de existência e log estruturado, cobrindo DPROD-01..03.

## must_haves
<truths>
- `alterarStatusProduto(idproduto, ativo)` verifica existência com `SELECT 1 FROM produto WHERE idproduto = $1 LIMIT 1` e lança `NotFoundException` (404) se não existir.
- Com `ativo === false`: UPDATE seta `statusproduto = false` e `vendeproduto = false` (DPROD-01).
- Com `ativo === true`: UPDATE seta `statusproduto = true` e `vendeproduto = true` (DPROD-02).
- O UPDATE inclui `idusuarioalteracao = $n` (= `ATHOS_SISTEMA_USUARIO_ID`) e tem alvo exclusivamente a tabela `produto`.
- NENHUM caminho de código emite `DELETE` — toggle é sempre UPDATE (DPROD-03).
- Log estruturado: operação `deactivate` (ativo=false) ou `reactivate` (ativo=true), com `idproduto` e `idusuario`.
- `client.release()` chamado em `finally` em todos os caminhos.
- Testes cobrem: deactivate seta ambos false; reactivate seta ambos true; produto inexistente → 404; nenhum DELETE emitido.
</truths>

<threat_model>
**Threats:**
- DELETE físico de produto quebrando integridade referencial (Tampering/Destruction): mitigado — método só emite UPDATE; teste asserindo ausência de DELETE; constraint de projeto "Produto: Nunca apagar fisicamente" respeitada (DPROD-03).
- Toggle de status em tabela não autorizada (Tampering/Elevation): mitigado — `UPDATE produto SET ...` com nome de tabela literal.
- Coerção incorreta de `ativo` permitindo estado parcial (Integrity): mitigado — o controller (Plan 04) tipa `ativo` como boolean via DTO `AlterarStatusProdutoDto` com `@IsBoolean`; o service usa o mesmo valor para `statusproduto` e `vendeproduto`, garantindo consistência.
**Residual risks:** Nenhum risco relevante — operação idempotente e restrita a dois campos booleanos de uma única tabela.
</threat_model>

- .planning/phases/33-api-de-escrita-de-produto/33-RESEARCH.md — Pattern 5 (Toggle de Status) e D-07

<action>
Adicionar método `async alterarStatusProduto(idproduto: number, ativo: boolean): Promise<{ idproduto: number; ativo: boolean }>` ao `AthosProdutoService`. Reutilizar `getPool()`.

1. `const sistemaUsuarioId = Number(process.env.ATHOS_SISTEMA_USUARIO_ID)`.
2. `pool.connect()` → `client`, abrir `try`.
3. Verificar existência: `SELECT 1 FROM produto WHERE idproduto = $1 LIMIT 1` com `[idproduto]`; se `rows.length === 0` lançar `NotFoundException(\`Produto ${idproduto} nao encontrado\`)`.
4. Executar exatamente um UPDATE: `UPDATE produto SET statusproduto = $1, vendeproduto = $1, idusuarioalteracao = $2 WHERE idproduto = $3` com `[ativo, sistemaUsuarioId, idproduto]`. O mesmo `$1` (`ativo`) aplica-se a `statusproduto` e `vendeproduto` (D-07). NUNCA emitir DELETE (DPROD-03).
5. `const operacao = ativo ? "reactivate" : "deactivate"`. `this.logger.log(\`${operacao} idproduto=${idproduto} idusuario=${sistemaUsuarioId}\`)`. Retornar `{ idproduto, ativo }`.
6. `catch (err)`: re-lançar se `instanceof NotFoundException`; senão `InternalServerErrorException("Erro ao alterar status do produto no Athos")`.
7. `finally { client.release(); }`.
</action>

- O UPDATE seta `statusproduto = $1` e `vendeproduto = $1` (mesmo placeholder) e inclui `idusuarioalteracao`.
- Nenhuma string no método contém `DELETE`.
- `alterarStatusProduto` chama `client.release()` em `finally`.
- `cd apps/backend && npx tsc --noEmit` exits 0.
