# Fase 999.1: Relatório de Code Review

**Revisado:** 2026-06-22
**Profundidade:** standard
**Arquivos revisados:** 30 (de fonte; demais filtrados)
**Status:** issues_found

## Resumo

Fase de segurança crítica: aceita upload de templates HTML/Handlebars arbitrários, renderiza via Puppeteer e protege escritas admin por API key + cookie de sessão. A arquitetura de defesa em profundidade está bem pensada em vários pontos — `shouldAllowRequest` bloqueia toda a rede do Puppeteer (anti-SSRF sólido), `timingSafeEqual` é usado em todas as comparações de segredos, `knownHelpersOnly:true` restringe os helpers Handlebars, e as chaves de admin de fato só existem no servidor (sem prefixo `NEXT_PUBLIC_`).

Porém a revisão encontrou falhas que comprometem a postura de segurança real do deploy:

1. **Gate de senha do frontend "fail-open"** combinado com a **ausência total das variáveis de gate (`ADMIN_API_KEY`, `CONFIG_PANEL_PASSWORD`) no `stack.env.example`** — em um deploy seguindo o exemplo, o painel admin fica acessível sem senha e o backend rejeita as escritas com 401 silencioso (ou, se o operador definir só `ADMIN_API_KEY`, o painel continua sem senha).
2. A "camada extra de sanitização" via `sanitize-html` é **efetivamente inócua** (`allowedTags:false` permite todas as tags), de modo que a única defesa real do upload é o conjunto de regex `DANGEROUS_PATTERNS`, que tem lacunas conhecidas.
3. A invariante "no máximo um template ativo" pode quebrar sob concorrência por falta de bloqueio/serialização.

As seções abaixo detalham cada achado com correção concreta.

## Critical Issues

### CR-01: Gate de senha do painel admin é "fail-open" e as variáveis de proteção não existem no env de exemplo

**File:** `apps/frontend/src/lib/admin-session.ts:84-88` + `deploy/stack.env.example` (ausência) + `apps/backend/src/modules/app.module.ts:23-35`

**Issue:** `requireAdminSession` retorna `true` (libera) quando `CONFIG_PANEL_PASSWORD` não está definida:
```ts
export function requireAdminSession(req: NextRequest): boolean {
  if (!process.env.CONFIG_PANEL_PASSWORD) return true; // <- fail-open
  ...
}
```
Nenhuma das três variáveis que sustentam a proteção desta fase — `ADMIN_API_KEY`, `CONFIG_PANEL_PASSWORD`, `CONFIG_PANEL_SESSION_SECRET` — aparece em `deploy/stack.env.example` (confirmado por busca: "none in deploy"). Além disso, `ADMIN_API_KEY` **não** está em `REQUIRED_ENV_VARS` no backend, então o backend sobe normalmente sem ela.

Consequência num deploy que segue o `.env.example`:
- `CONFIG_PANEL_PASSWORD` indefinida → todas as rotas proxy `/api/admin/templates/*` liberam acesso sem nenhuma sessão (qualquer um que alcance a URL pode listar/subir/ativar/excluir/preview).
- `ADMIN_API_KEY` indefinida no backend → `adminBackendFetch` envia `x-admin-api-key: ""`, e `AdminAuthGuard` lança 401 (`!expected`). O painel fica visível e operável no frontend, mas toda ação falha com erro genérico de backend — UX quebrada e, pior, a percepção de que "há proteção" é falsa.
- Se o operador definir só `ADMIN_API_KEY` (para destravar o backend) e esquecer `CONFIG_PANEL_PASSWORD` → painel **100% aberto** a quem alcançar a rota, pois o backend agora aceita a chave injetada server-side e o frontend não exige sessão.

**Fix:**
1. Adicionar ao `deploy/stack.env.example`, na seção Backend e Frontend, com instruções:
```env
# Proteção do painel de templates de PDF (999.1) — OBRIGATÓRIAS
# ADMIN_API_KEY: credencial do AdminAuthGuard (backend). Gere com: openssl rand -hex 32
ADMIN_API_KEY=
# CONFIG_PANEL_PASSWORD: senha do gate da tela /configuracoes/templates (frontend, server-side)
CONFIG_PANEL_PASSWORD=
# CONFIG_PANEL_SESSION_SECRET: segredo HMAC do cookie de sessão (opcional; cai em ADMIN_API_KEY)
CONFIG_PANEL_SESSION_SECRET=
```
2. Tornar o gate fail-closed por padrão ou, no mínimo, logar um aviso crítico de startup quando `CONFIG_PANEL_PASSWORD` estiver ausente em produção:
```ts
export function requireAdminSession(req: NextRequest): boolean {
  if (!process.env.CONFIG_PANEL_PASSWORD) {
    if (process.env.NODE_ENV === "production") return false; // fail-closed em prod
    return true;
  }
  const token = req.cookies.get(ADMIN_SESSION_COOKIE)?.value;
  return isValidSessionToken(token);
}
```
3. Considerar adicionar `ADMIN_API_KEY` a `REQUIRED_ENV_VARS` (app.module.ts) para falhar no boot se o módulo de templates estiver ativo sem a chave.

### CR-02: A camada de sanitização adicional é inócua — `allowedTags:false` permite todas as tags

**File:** `apps/backend/src/modules/pdf-templates/pdf-templates.service.ts:64-76, 167-175`

**Issue:** A validação se apresenta como dupla defesa: regex `DANGEROUS_PATTERNS` **mais** uma checagem de divergência via `sanitize-html`. Porém a configuração usada anula a segunda camada:
```ts
const sanitized = sanitizeHtml(raw, {
  allowedTags: false,        // <- permite TODAS as tags
  allowedAttributes: false,  // <- permite TODOS os atributos
  disallowedTagsMode: "discard",
});
```
Em `sanitize-html`, `allowedTags === false` faz `tagAllowed()` retornar `true` para qualquer tag (confirmado em `node_modules/sanitize-html/index.js:124-125`). Com tags e atributos todos permitidos, `sanitized.length` fica essencialmente igual a `raw.length`, então `hasSignificantDivergence` (limiar de 30% removido) praticamente nunca dispara. A "segunda camada" não detecta nada que a primeira já não tenha barrado — é falsa sensação de defesa em profundidade. O comentário no código afirma o contrário ("Sinal adicional...trate como suspeito também"), o que induz manutenção futura ao erro.

**Fix:** Ou remover a checagem (e o comentário) para não mascarar a real superfície de proteção, ou configurar `sanitize-html` com uma allow-list real de tags/atributos compatível com os templates esperados e comparar contra ela. Exemplo de detecção real:
```ts
const sanitized = sanitizeHtml(raw, {
  allowedTags: ["html","head","body","style","meta","title","div","span","p","h1","h2",
    "table","thead","tbody","tr","th","td","img","svg","path","b","br","header"],
  allowedAttributes: { "*": ["class","style","id","lang","charset","content","name",
    "viewbox","xmlns","d","fill","fill-rule","src","alt","width","height"] },
  disallowedTagsMode: "discard",
});
if (this.hasSignificantDivergence(raw, sanitized)) { /* rejeita */ }
```

### CR-03: `DANGEROUS_PATTERNS` não cobre `<base>`, `<meta http-equiv=refresh>`, `data:` em `<a>`/`<use>` e `style="expression(...)"`

**File:** `apps/backend/src/modules/pdf-templates/pdf-templates.service.ts:18-27`

**Issue:** A allow-by-default da validação confia exclusivamente nestes regex. Probes confirmaram que passam sem rejeição:
- `<base href="http://evil/">` — não bloqueado (reescreveria URLs relativas).
- `<meta http-equiv="refresh" content="0;url=...">` — não bloqueado.
- `<style>@import url(...)</style>` — não bloqueado.
- `<svg><use href="data:image/svg+xml,...">` — não bloqueado.
- `<div style="width:expression(alert(1))">` — não bloqueado.

O impacto é parcialmente mitigado pelo bloqueio total de rede do Puppeteer (`shouldAllowRequest`) e pelo escaping de Handlebars nas expressões `{{ }}`, então a exfiltração via rede é neutralizada. Mas continua sendo uma quebra da postura "rejeitar conteúdo perigoso" prometida pela D-02: `<base>`/`<meta refresh>`/`data:` URIs são vetores legítimos de manipulação do documento renderizado e devem ser barrados na origem (o template é confiável uma vez ativo e renderizado para clientes reais).

**Fix:** Estender a lista e/ou substituir pela sanitização com allow-list real (ver CR-02):
```ts
const DANGEROUS_PATTERNS: RegExp[] = [
  /<script[\s>]/i, /<\/script/i,
  /\son\w+\s*=/i,
  /javascript:/i, /vbscript:/i, /data:\s*text\/html/i,
  /<iframe[\s>]/i, /<object[\s>]/i, /<embed[\s>]/i,
  /<base[\s>]/i,
  /<meta[^>]+http-equiv\s*=\s*["']?refresh/i,
  /expression\s*\(/i,
  /@import/i,
  /file:/i,
];
```

### CR-04: Swap atômico do template ativo não impede dois ativos sob concorrência

**File:** `apps/backend/src/modules/pdf-templates/pdf-templates.service.ts:106-120` + `prisma/schema.prisma:291-302`

**Issue:** A invariante "no máximo um `isActive=true`" é declarada como garantida "por transação no serviço, não por constraint de banco" (schema.prisma:288-290). Mas a transação usada é uma sequência de operações sem isolamento serializável nem bloqueio:
```ts
const [, updated] = await this.prisma.$transaction([
  this.prisma.pdfTemplate.updateMany({ data: { isActive: false } }),
  this.prisma.pdfTemplate.update({ where: { id }, data: { isActive: true } }),
]);
```
Sob duas chamadas `activate` concorrentes (A e B), no nível de isolamento padrão (`READ COMMITTED`) ambas podem zerar todos os ativos e depois cada uma ativar o seu id, resultando em **dois** templates ativos. Como `getActiveSource` usa `findFirst({ where: { isActive: true } })` (repository.ts:21), o template servido aos clientes passa a ser não-determinístico. Além disso, `activate` busca o template (`findUnique`, linha 107) fora da transação — uma exclusão concorrente entre a leitura e a transação levaria a um `update` em id inexistente.

O painel é multiusuário (vários sócios), então concorrência não é hipotética.

**Fix:** Aplicar uma constraint de banco real (partial unique index) como fonte de verdade, e/ou serializar. Migração:
```sql
CREATE UNIQUE INDEX "PdfTemplate_single_active_idx"
  ON "PdfTemplate" ("isActive") WHERE "isActive" = true;
```
Com o índice parcial, a 2ª ativação concorrente falha com violação de unicidade (tratável como 409). Alternativamente, usar `$transaction(fn, { isolationLevel: Prisma.TransactionIsolationLevel.Serializable })` e mover o `findUnique` para dentro da transação.

## Warnings

### WR-01: Reuso de `ADMIN_API_KEY` como segredo de assinatura do cookie acopla rotação e expõe a chave a mais um uso

**File:** `apps/frontend/src/lib/admin-session.ts:23-33`

**Issue:** `sessionSecret()` cai para `ADMIN_API_KEY` (e depois `INTERNAL_API_KEY`) quando `CONFIG_PANEL_SESSION_SECRET` não está definida. Isso significa que rotacionar a `ADMIN_API_KEY` invalida silenciosamente todas as sessões existentes, e o mesmo segredo passa a ter dupla finalidade (auth de backend e assinatura de cookie de frontend), ampliando o raio de impacto se um for comprometido. Se ambos `CONFIG_PANEL_SESSION_SECRET`, `ADMIN_API_KEY` e `INTERNAL_API_KEY` estiverem ausentes, `sessionSecret()` retorna `""` e o HMAC passa a usar chave vazia — assinatura previsível.

**Fix:** Exigir `CONFIG_PANEL_SESSION_SECRET` (ou ao menos falhar/avisar quando o segredo resolvido for `""`):
```ts
function sessionSecret(): string {
  const s = process.env.CONFIG_PANEL_SESSION_SECRET
    ?? process.env.ADMIN_API_KEY ?? process.env.INTERNAL_API_KEY ?? "";
  if (!s) throw new Error("Sessão admin sem segredo de assinatura configurado.");
  return s;
}
```

### WR-02: Upload e preview não validam/limitam `name` nem o `source` no proxy do frontend

**File:** `apps/frontend/src/app/api/admin/templates/route.ts:22-43` e `apps/backend/src/modules/pdf-templates/pdf-templates.controller.ts:62-66`

**Issue:** O controller backend recebe `body: { name: string; source: string }` sem `class-validator`/DTO (o projeto usa `class-validator`, dependência presente). `name` é repassado direto a `create` e usado em `buildUniqueSlug`. Um `name` vazio gera slug `template-<suffix>` (ok), mas `name` ausente (undefined) seria gravado como `null`/erro Prisma só em runtime, e não há limite de tamanho de `name`. O `validateUpload` cobre `source`, mas não há checagem de que `source` é string (se vier `source: 123`, `Buffer.byteLength(123)` lança TypeError não-tratado → 500).

**Fix:** Adicionar um DTO com validação no controller:
```ts
class CreateTemplateDto {
  @IsString() @IsNotEmpty() @MaxLength(120) name!: string;
  @IsString() @IsNotEmpty() source!: string;
}
upload(@Body() body: CreateTemplateDto) { return this.pdfTemplatesService.create(body); }
```
e garantir `ValidationPipe` global (verificar em `main.ts`, fora do escopo desta leitura).

### WR-03: Preview aceita `source` arbitrário SEM passar por `validateUpload`

**File:** `apps/backend/src/modules/pdf-templates/pdf-templates.controller.ts:89-108`

**Issue:** O endpoint `POST /pdf-templates/preview` renderiza `body.source` diretamente via `renderPreviewPdf` sem chamar `validateUpload`. Embora o render seja hardened (rede bloqueada, helpers restritos), isso significa que a única barreira contra `source` perigoso no preview é o Puppeteer hardened — não a validação de upload. Como o preview é uma ação admin autenticada e o render é sandboxed na rede, o risco é baixo, mas é inconsistente com a postura "rejeitar conteúdo perigoso antes de renderizar" e permite que um admin pré-visualize conteúdo que o `create` recusaria, gerando confusão (preview funciona, salvar falha).

**Fix:** Rodar `validateUpload(source)` também no caminho de preview com `source` explícito (não para `templateId`, que já foi validado no create):
```ts
const source = body.templateId
  ? await this.pdfTemplatesService.getSource(body.templateId)
  : body.source ? this.pdfTemplatesService.validateUpload(body.source) : undefined;
```

### WR-04: `seed-pdf-templates.ts` pode reativar `colorful` e desativar a escolha do usuário a cada deploy

**File:** `apps/backend/scripts/seed-pdf-templates.ts:65-79` + `bootstrap-runtime.sh:25-27`

**Issue:** O seed roda em todo boot (`bootstrap-runtime.sh`) e faz `upsert` com `update: { isActive: preset.isActive }`, forçando `colorful.isActive=true` e `minimal/classic.isActive=false`. Se o admin tiver ativado `minimal` (ou um template CUSTOM), o próximo deploy:
- Re-marca `colorful` como ativo (update force `isActive:true`).
- Não toca no CUSTOM ativo do usuário → resultado: **dois ativos** (colorful + o custom do usuário), quebrando a invariante e tornando `getActiveSource` não-determinístico (mesmo problema de CR-04, agora disparado por deploy rotineiro).

**Fix:** No `update`, não sobrescrever `isActive` (preservar a escolha do operador); só definir `isActive` no `create` (primeira semeadura):
```ts
update: { name: preset.name, source }, // sem isActive
create: { name, slug, source, kind: PRESET, isActive: preset.isActive },
```
E garantir, na primeira semeadura, que apenas um fique ativo (ou aplicar o índice parcial de CR-04).

### WR-05: `remove()` permite excluir presets, deixando o seed/bootstrap inconsistente

**File:** `apps/backend/src/modules/pdf-templates/pdf-templates.service.ts:152-165`

**Issue:** `remove` só impede excluir o template ativo; presets (`kind=PRESET`) podem ser excluídos como qualquer custom. O comentário admite isso intencionalmente, mas combinado com o seed idempotente por `slug`, excluir um preset faz o próximo deploy recriá-lo (upsert por slug) — exclusão "não cola". Para o usuário, o preset some e reaparece após deploy, comportamento confuso. Pior: se o preset excluído era o `colorful` (default), o seed o recria com `isActive=true`, podendo reintroduzir o problema de duplo-ativo de WR-04.

**Fix:** Bloquear exclusão de `kind=PRESET` (presets são fixos, gerenciados pelo seed), ou marcar presets como "ocultáveis" em vez de excluíveis:
```ts
if (template.kind === PdfTemplateKind.PRESET) {
  throw new BadRequestException("Modelos prontos não podem ser excluídos.");
}
```

### WR-06: `Content-Length` setado com número em vez de string e antes do `send`

**File:** `apps/backend/src/modules/pdf-templates/pdf-templates.controller.ts:106-108`

**Issue:** `res.setHeader("Content-Length", pdfBuffer.length)` passa um `number`. Express normalmente coage, mas o contrato de `setHeader` é `string | string[] | number` apenas em versões recentes; em alguns adapters isso gera warning ou header malformado. Além disso, `res.send(Buffer)` já define `Content-Length` automaticamente, então o set manual é redundante e pode conflitar.

**Fix:** Remover a linha de `Content-Length` (deixe o `res.send` cuidar) ou converter para string. Recomendado remover:
```ts
res.setHeader("Content-Type", "application/pdf");
res.send(pdfBuffer);
```

### WR-07: `formatNumber`/`formatCurrency` recebem strings dos presets de teste mas `number` em produção — coerção silenciosa

**File:** `apps/backend/src/modules/quotes/quotes-pdf-storage.service.ts:157-159, 397-403`

**Issue:** Em `renderHtml`, `quantidadeitem`, `valoritem` etc. são pré-formatados para string via `formatNumber`/`formatCurrency`. `formatCurrency` faz `Number(value ?? 0)` — se o payload trouxer `valoritem: "abc"` (dado externo do Athos/PDV), `Number("abc")` é `NaN` e `Intl.NumberFormat` renderiza "R$ NaN" no PDF do cliente, sem erro. `formatNumber` não tem o `Number()` defensivo e produziria `NaN` igualmente. Não há validação de que os campos numéricos do payload são realmente numéricos.

**Fix:** Sanitizar/validar os valores numéricos do payload na fronteira (coerção explícita com fallback):
```ts
private formatCurrency(value: unknown): string {
  const n = Number(value); const safe = Number.isFinite(n) ? n : 0;
  return new Intl.NumberFormat("pt-BR", { style: "currency", currency: "BRL" }).format(safe);
}
```

### IN-01: `ExpressResponse = any` desabilita type-safety no controller

**File:** `apps/backend/src/modules/pdf-templates/pdf-templates.controller.ts:2-3`

### IN-02: Aviso de "fontes web" enganoso com rede bloqueada

### IN-03: `KNOWN_HELPERS` vazio duplicado entre service de teste e produção como objeto literal

**Issue:** A configuração de compile hardened (`knownHelpers: {}`, `knownHelpersOnly: true`, etc.) é replicada manualmente em vários lugares (service e dois testes). Divergência futura (alguém habilitar um helper no service e esquecer o teste) não seria detectada.

**Fix:** Extrair a config de compile para uma constante exportada compartilhada e reusá-la nos testes.

### IN-04: `abrirPdfBlob` usa `URL.revokeObjectURL` com timeout fixo de 60s

**File:** `apps/frontend/src/app/configuracoes/templates/templates-manager.tsx:131-136`

**Issue:** O blob URL é revogado após 60s fixos. Se o PDF for grande e a aba demorar a carregar, pode ser revogado antes do uso; se o usuário fechar antes, o objeto vive 60s desnecessariamente. Pequeno vazamento/risco de borda, não funcional na maioria dos casos.

**Fix:** Aceitável como está para uso interno; opcionalmente revogar no evento de carga da nova aba.

### IN-05: Mensagens de log/erro inconsistentes com acentuação (mistura "Nao"/"Não")

**Fix:** Padronizar para português correto ("Não informado", "Sem observações") nos valores que chegam ao PDF.
