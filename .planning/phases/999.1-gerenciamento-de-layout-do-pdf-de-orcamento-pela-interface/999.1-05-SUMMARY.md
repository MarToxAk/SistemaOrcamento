# Phase 999.1 Plan 05: Módulo Backend de Gerenciamento de Templates PDF — Summary

**One-liner:** PdfTemplatesService (validação que REJEITA HTML perigoso, swap atômico do ativo via transação) + PdfTemplatesController (5 endpoints `@AdminOnly()`: listar/upload/ativar/excluir/preview) + novo `renderPreviewPdf` no QuotesPdfStorageService para preview server-side sem persistir, todos consumindo a fundação dos Planos 01-04.

## Resolução do Checkpoint

O plano continha um `checkpoint:human-verify` (gate `blocking-human`) antes do `npm install sanitize-html`, pois o RESEARCH marcou o pacote como SUS apenas pela heurística "too-new" (último patch 2026-06-10). O orquestrador verificou independentemente `sanitize-html@2.17.5` no npm: mantenedores são todos `@apostrophecms.com` (equipe oficial ApostropheCMS, incluindo Tom Boutell), repositório `github.com/apostrophecms`, cadência de patch mensal estável, milhões de downloads semanais. A flag "too-new" foi confirmada como falso positivo. Checkpoint **aprovado** — execução prosseguiu sem re-apresentar o gate.

## Architecture Delivered

```
PdfTemplatesService
  ├── validateUpload(raw) → REJEITA (BadRequestException) se:
  │     - >512KB (T-999.1-15)
  │     - script/on*=/javascript:/iframe/object/embed/file: (T-999.1-14)
  │     - sanitize-html remove >30% do conteúdo (sinal adicional de suspeita)
  │   → preserva o source ORIGINAL intacto quando aprovado (CSS/Handlebars legítimos)
  ├── create({name, source}) → valida + grava kind=CUSTOM, isActive=false
  ├── activate(id) → prisma.$transaction([updateMany(isActive:false), update(isActive:true)]) (D-04)
  ├── list() → PdfTemplateSummary[] (sem campo source)
  └── remove(id) → impede excluir o template ATIVO (NotFoundException se inexistente)

PdfTemplatesController (prefixo "pdf-templates", todos @AdminOnly())
  ├── GET    /                 → list()
  ├── POST   /                 → upload({name, source}) → service.create
  ├── PATCH  /:id/activate     → activate(id)
  ├── DELETE /:id              → remove(id)
  └── POST   /preview          → renderiza PDF via QuotesPdfStorageService.renderPreviewPdf
                                  (payload MOCK + source explícito) — Content-Type application/pdf,
                                  SEM persistir nada (sem MinIO putObject, sem Prisma write)

QuotesPdfStorageService.renderPreviewPdf(payload, templateSource)  [NOVO — D-08]
  └── renderHtml(payload, templateSource) → renderPdfBuffer(html)
      (exatamente o mesmo caminho hardened de generateAndStore, sem o putObject final)

PdfTemplatesModule
  └── imports: QuotesModule (reusa QuotesPdfStorageService — decisão única, resolve W3)
  └── DatabaseModule NÃO importado explicitamente (já é @Global())
  └── registrado em app.module.ts
```

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 2 - Funcionalidade crítica faltante] `renderPreviewPdf` não existia em QuotesPdfStorageService**
- **Found during:** Task 2 (preparação do endpoint de preview)
- **Issue:** O plano (e o `key_links` do frontmatter) descreve "preview chama o render com templateSource explícito" via `renderPreviewHtml|templateSource`, mas nenhum método público combinava `renderHtml` + `renderPdfBuffer` sem persistir — `renderPdfBuffer` era privado e `generateAndStore` sempre grava no MinIO.
- **Fix:** Adicionado `renderPreviewPdf(payload, templateSource)` público em `QuotesPdfStorageService`, reutilizando exatamente o mesmo caminho hardened (`renderHtml` + `renderPdfBuffer`) sem o passo de `putObject`. Mantém T-999.1-17 (preview usa render hardened, nunca um caminho alternativo).
- **Files modified:** quotes-pdf-storage.service.ts
- **Commit:** 1468e54

**2. [Rule 3 - Blocking] `@types/express` ausente do projeto**
- **Found during:** Task 2 (criação do controller com `@Res()`)
- **Issue:** `import type { Response } from "express"` causaria erro de tipos — o projeto só tem a lib `express` como dependência transitiva, sem `@types/express` instalado.
- **Fix:** Replicado o padrão já existente em `cobranca.controller.ts`: `type ExpressResponse = any` com `eslint-disable-next-line @typescript-eslint/no-explicit-any`.
- **Files modified:** pdf-templates.controller.ts
- **Commit:** 1468e54

**3. [Rule 3 - Blocking] `Test.createTestingModule` falhou ao resolver `ConfigService` para `AdminAuthGuard` em teste unitário do controller**
- **Found during:** Task 2 (primeira execução dos testes do controller)
- **Issue:** Mesmo em teste unitário de controller (sem montar app HTTP), o NestJS `TestingModule` resolve o grafo de dependências dos guards anexados via `@UseGuards` metadata, e `AdminAuthGuard` requer `ConfigService`.
- **Fix:** Adicionado `{ provide: ConfigService, useValue: { get: jest.fn() } }` aos providers do módulo de teste.
- **Files modified:** pdf-templates.controller.test.ts
- **Commit:** 1468e54
