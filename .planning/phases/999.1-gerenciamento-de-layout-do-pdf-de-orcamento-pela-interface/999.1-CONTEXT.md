# Phase 999.1: Gerenciamento de layout do PDF de orçamento pela interface - Context

**Gathered:** 2026-06-22
**Status:** Ready for planning

<domain>
## Phase Boundary

Permitir que o layout/template do PDF de orçamento seja trocado em runtime pelo próprio sistema (usuário final), **sem editar código, sem commit/git, sem reiniciar o servidor**. Extensão das fases 35 (backend white-label) e 36 (frontend white-label). Inclui: (1) backend para armazenar/selecionar/subir templates em runtime; (2) tela de configuração no frontend para escolher entre modelos prontos e subir um `.hbs` customizado, com preview; (3) 3 modelos prontos iniciais; (4) correção do contato BomCusto ainda hardcoded no template; (5) **hardening de segurança obrigatório** pelo upload de HTML/Handlebars arbitrário.

Fora de escopo: editor visual WYSIWYG (montar layout arrastando blocos); multi-tenant com múltiplos templates ativos simultâneos no mesmo deploy.

</domain>

<decisions>
## Implementation Decisions

### Nível de Customização (A → c)

- **D-01:** Dois caminhos coexistem: (a) **galeria de presets prontos** — usuário escolhe um layout e marca como ativo; (b) **upload de template `.hbs`/HTML customizado** pelo próprio usuário. Ambos disponíveis na mesma tela.
- **D-02:** O upload de HTML/Handlebars arbitrário cria superfície de segurança alta. **Mitigações OBRIGATÓRIAS (requer `/gsd-secure-phase`):** allowlist de helpers Handlebars (bloquear helpers/expressões perigosas), sanitização/validação do HTML recebido, bloqueio de SSRF e fetch de recursos externos durante o render no Puppeteer (interceptar/negar requisições de rede não permitidas), hardening do Puppeteer (rever `--no-sandbox` atual, isolar contexto, timeout, sem acesso a `file://`). Esta é a decisão pivotal que define o esforço da fase.

### Controle de Acesso (B → b)

- **D-03:** A tela de gerenciamento de template e os endpoints de escrita (upload, selecionar ativo, excluir) são protegidos por **senha/chave de admin**, porque o app não possui sistema de login/perfis de usuário hoje. A leitura/geração de PDF permanece como está. Mecanismo exato (variável de ambiente `ADMIN_*` / senha dedicada, header extra além do `INTERNAL_API_KEY`) é Claude's Discretion / a definir em research.

### Armazenamento dos Templates

- **D-04:** Templates (presets + uploads) e a marcação de "qual está ativo" são persistidos em storage editável em runtime. Candidatos: tabela no banco (Prisma — já existe `quoteDocument`) ou MinIO (já usado para os PDFs). Single-tenant: **um template ativo por deploy**. Escolha banco × MinIO fica para research/planner decidir com base em backup, versionamento e simplicidade.

### Troca em Runtime

- **D-05:** `renderHtml()` em `quotes-pdf-storage.service.ts` (hoje lê arquivo via `readFileSync` a cada geração — D-05 da fase 35) passa a buscar o **template ativo do storage** a cada geração. Mantém a cadeia de fallback existente (`EMPRESA_PDF_TEMPLATE_PATH` → `quote-default.hbs` → string embutida) como rede de segurança quando não há template ativo selecionado.

### Modelos Prontos Iniciais (C → 3 modelos)

- **D-06:** Três layouts de partida: (1) **colorido atual** (`quote-default.hbs` existente), (2) **minimalista P&B**, (3) **clássico**. Todos já com o contato dehardcoded (D-07).

### Contato Hardcoded

- **D-07:** Telefones, email (`orcamento@bomcustoilhabela.com.br`) e Instagram (`@bomcustopapelaria`) ainda hardcoded em `quote-default.hbs` (linhas ~175-179) viram dados configuráveis (variáveis de empresa / campos), eliminando o vazamento de branding remanescente das fases 35/36.

### Preview

- **D-08:** A tela permite **preview do PDF antes de ativar/aplicar** um template (recomendado para evitar ativar um layout quebrado em produção). Forma exata (render server-side com dados de exemplo, abrir em nova aba) é Claude's Discretion.

### Claude's Discretion

- Banco (Prisma) × MinIO para armazenar templates.
- Mecanismo exato da credencial de admin (env var, header, hash de senha).
- Forma do preview (dados de exemplo vs orçamento real).
- Estrutura dos endpoints REST (listar/ativar/upload/excluir/preview).

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets

- **`renderHtml()` lê template por requisição** — `readFileSync` a cada geração (fase 35 D-05); trocar a fonte para o storage ativo dá a troca em runtime sem restart "de graça".
- **Cliente MinIO já configurado** em `quotes-pdf-storage.service.ts` (`buildMinioClient`, `putObject`, `getObject`, `bucketExists`/`makeBucket`) — reutilizável para armazenar templates se optar por MinIO.
- **Prisma + `quoteDocument`** — banco já em uso; tabela de templates é caminho natural se optar por banco.
- **`InternalAuthGuard`** (`security/internal-auth.guard.ts`) — padrão de guard por header; base para a credencial de admin (D-03).
- **Cadeia de fallback de template** (`EMPRESA_PDF_TEMPLATE_PATH` → `.hbs` → string) — manter como rede de segurança quando não há template ativo.
- **Variáveis de empresa no contexto Handlebars** (`empresaNome`, `empresaCnpj`, `empresaEndereco`, `empresaLogoUrl`, `empresaCor`) já passadas ao template — base para dehardcode do contato (D-07).

### Established Patterns

- **Handlebars + Puppeteer** — `Handlebars.compile(src)(ctx)` → `page.setContent(html)` → `page.pdf()`. O hardening de segurança incide exatamente aqui.
- **Guards globais** via `APP_GUARD` (`InternalAuthGuard`, `ThrottlerGuard`) em `app.module.ts`.
- **Rotas proxy do frontend** em `apps/frontend/src/app/api/**/route.ts` injetam `INTERNAL_API_KEY` server-side — frontend nunca expõe a chave.

### Integration Points

- **Frontend sem login/perfis** — não há sessão de usuário; por isso D-03 usa credencial de admin dedicada para a tela de gerenciamento.
- **Novo módulo/endpoints** no backend para CRUD de templates + seleção do ativo + preview, protegidos pela credencial de admin.
- **Nova tela** no frontend (galeria de presets + upload + preview), consumindo os endpoints via rota proxy.

</code_context>

<specifics>
## Specific Ideas

- Os 3 presets iniciais devem todos consumir as mesmas variáveis Handlebars já documentadas no topo de `quote-default.hbs`, para que troquem de layout sem perder dados.
- Preview idealmente renderiza com um orçamento de exemplo (mock) para não depender de um orçamento real existente.
- Ao desativar/trocar template, manter o anterior salvo (não destruir) para rollback rápido.

</specifics>
