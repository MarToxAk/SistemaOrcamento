## Summary

Esta fase realiza WL-01/WL-03 (deferidos das fases 35/36): trocar o template do PDF de orçamento em runtime pela interface, com presets prontos e upload de HTML/Handlebars arbitrário. O ponto central de mudança é `QuotesPdfStorageService.renderHtml()` / `renderPdfBuffer()` — hoje resolve template via `readFileSync` por requisição e renderiza com Puppeteer usando `--no-sandbox`. A decisão pivotal (D-02) é permitir HTML arbitrário, o que cria uma superfície de RCE/SSRF/exfiltração significativa.

A boa notícia é que o codebase já tem todos os primitivos necessários: cliente MinIO configurado, Prisma com migrations, `InternalAuthGuard` como modelo de credencial, e rotas proxy do frontend que injetam a chave server-side. O esforço concentra-se quase inteiramente no **hardening de segurança**, não em infraestrutura nova. Handlebars 4.7.9 (instalado) já bloqueia acesso a protótipo por padrão e está corrigido contra a CVE-2019-19919, mas templates de usuário ainda exigem `knownHelpersOnly:true` com allowlist de helpers e `allowCallsToHelperMissing:false`. A neutralização de SSRF/exfiltração se faz no Puppeteer via `page.setRequestInterception(true)` abortando **toda** requisição de rede exceto o documento inline.

**Atenção crítica de regressão:** o preset "colorido" atual (`quote-default.hbs`) depende de recursos EXTERNOS — Google Fonts, jsDelivr (bootstrap-icons). A mitigação de SSRF (bloquear toda rede) vai quebrar visualmente esse preset a menos que fontes/ícones sejam inlinados ou self-hosted. Isso precisa ser tratado na mesma fase.

**Primary recommendation:** Persistir templates no **banco (Prisma)** com um modelo `PdfTemplate` + ponteiro de ativo; aplicar Handlebars hardened (`knownHelpersOnly`, sem helpers perigosos, sem proto access) + `sanitize-html` na validação de upload + interceptação total de rede no Puppeteer; gatear escrita/preview com um **segundo header de admin** (`x-admin-api-key` validado por `timingSafeEqual`, mesmo padrão do `InternalAuthGuard`); inlinar fontes/ícones nos 3 presets para sobreviver ao bloqueio de rede.

### Pattern 1: Compilação Handlebars hardened para template não confiável
**What:** Compilar o template do usuário desabilitando helpers desconhecidos e acesso a protótipo.
**When to use:** SEMPRE que o `templateSource` vier de upload (não vale para a string embutida confiável, mas aplicar uniformemente simplifica).
**Example:**
```typescript
// Source: handlebarsjs.com/api-reference/runtime-options.html [CITED]
// Helpers permitidos = exatamente os que os 3 presets usam (#if, #each, #unless são built-in
// de bloco e NÃO precisam estar em knownHelpers; knownHelpers cobre helpers nomeados/custom).
const KNOWN_HELPERS = {
  // built-ins de bloco já reconhecidos pelo compilador; liste aqui apenas helpers custom
  // que você REGISTRAR explicitamente. Não registre nenhum helper custom de I/O.
};

const template = Handlebars.compile(templateSource, {
  knownHelpers: KNOWN_HELPERS,
  knownHelpersOnly: true,        // qualquer helper fora da lista = erro de compilação
  strict: false,                 // manter false: presets usam {{#if cliente.telefone}} em campos ausentes
  noEscape: false,               // manter escaping de HTML nas expressões {{ }}
  // defaults 4.6+ já seguros (NÃO sobrescrever):
  //   allowProtoPropertiesByDefault: false
  //   allowProtoMethodsByDefault: false
  //   allowCallsToHelperMissing: false   <-- nunca true (habilita RCE)
});
```
**Nota:** `{{#if}}`, `{{#each}}`, `{{#unless}}`, `{{#with}}`, `lookup` são built-ins do compilador — `knownHelpersOnly:true` os mantém. O objetivo do allowlist é impedir que o template invoque qualquer helper custom perigoso que porventura tenha sido registrado globalmente. **Não registre `require`, helpers de fs/exec, nem `allowCallsToHelperMissing:true`.**

### Pattern 2: Bloqueio total de rede no Puppeteer (anti-SSRF / anti-exfiltração)
**What:** Abortar toda requisição de rede durante o render; só o documento inline é permitido.
**When to use:** Em `renderPdfBuffer()`, para qualquer render (presets e uploads).
**Example:**
```typescript
// Source: pptr.dev/api/puppeteer.page.setrequestinterception [CITED] + puppeteer.guide/posts/sandbox [CITED]
const page = await browser.newPage();
await page.setRequestInterception(true);
page.on("request", (req) => {
  // setContent não navega; o "document" inicial é about:blank/data — sem rede legítima necessária.
  // Bloqueia http(s) externos, file://, e qualquer recurso (font/img/css/xhr/fetch) =>
  // neutraliza SSRF para 169.254.169.254 (metadata), IPs internos, e exfiltração via beacon.
  const url = req.url();
  const isInlineDoc = url === "about:blank" || url.startsWith("data:");
  if (isInlineDoc) return req.continue();
  return req.abort();
});
await page.setContent(html, { waitUntil: "load", timeout: 30000 }); // reduzir de 60s p/ 30s
```
**Consequência (regressão):** com a rede bloqueada, Google Fonts e jsDelivr (bootstrap-icons) do `quote-default.hbs` NÃO carregam. Os 3 presets devem **inlinar** as fontes (base64 `@font-face` ou usar fontes do sistema/`fonts-liberation` já no Dockerfile) e substituir bootstrap-icons por SVG inline ou caracteres Unicode. Sem isso, o "colorido" sai com tipografia/ícones quebrados.

### Anti-Patterns to Avoid
- **Confiar no client para sanitizar/validar:** toda validação de HTML é server-side; o frontend pode até pré-avisar, mas a autoridade é o backend.
- **Manter `--no-sandbox` rodando como root com HTML não confiável:** combinação que transforma um bug do Chromium em comprometimento do host. Ver Pitfall 1.
- **Reusar `INTERNAL_API_KEY` para admin:** viola separação de privilégio (D-03 pede credencial dedicada).
- **`allowCallsToHelperMissing: true` ou registrar helpers de I/O:** caminho direto para RCE no template.
- **Bloquear rede sem inlinar fontes/ícones dos presets:** quebra visualmente o "colorido".
- **`waitUntil: "networkidle0"`:** com rede bloqueada nunca há "idle"; usar `"load"` (já é o caso) com timeout curto.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Sanitização de HTML | Regex p/ remover `<script>` | `sanitize-html` (allowlist) | Regex é trivialmente contornável (`<scr<script>ipt>`, eventos `on*`, `<svg onload>`); allowlist de tags/atributos é a abordagem correta |
| Bloqueio de SSRF/recursos | Parsear URLs do HTML manualmente | `page.setRequestInterception` + abort | O browser resolve URLs relativas, redirects, `@import`, `url()` em CSS — interceptar no nível de rede pega tudo |
| Restringir Handlebars | Parsear/regex o template | Opções de compile (`knownHelpersOnly`) | O parser do próprio Handlebars conhece a sintaxe; defaults 4.6+ já bloqueiam protótipo |
| Comparação de chave de admin | `provided === expected` | `crypto.timingSafeEqual` | Evita timing attack; já é o padrão do `InternalAuthGuard` |
| Sandbox do Chromium | Flags ad-hoc | Sandbox habilitado + seccomp + non-root | Reimplementar isolamento é inviável; usar o do Chrome |

**Key insight:** Renderizar HTML arbitrário em um browser headless é executar código não confiável. A defesa correta é em camadas (sanitização → compile restrito → rede bloqueada → sandbox do SO), nenhuma sozinha é suficiente.

## Common Pitfalls

### Pitfall 1: `--no-sandbox` + root + HTML não confiável (atual)
**What goes wrong:** O Dockerfile instala Chromium e roda como root; o serviço lança Puppeteer com `--no-sandbox --disable-setuid-sandbox`. Com HTML arbitrário, um exploit do renderizador vira execução de código no container como root.
**Why it happens:** `--no-sandbox` foi adicionado porque o sandbox do Chrome não sobe como root em container sem ajustes.
**How to avoid:** Adicionar usuário não-root no Dockerfile (`pptruser`), rodar o processo como ele, e habilitar o sandbox (`--shm-size=2g` + perfil seccomp do Chrome, evitando `SYS_ADMIN` amplo). Se manter `--no-sandbox` for inevitável no ambiente atual, então o bloqueio total de rede (Pattern 2) + sanitização + compile restrito tornam-se as defesas primárias — documentar explicitamente em `/gsd-secure-phase` que essa é a postura aceita e por quê. `[CITED: puppeteer.guide/posts/sandbox]`
**Warning signs:** `Running as root without --no-sandbox is not supported` nos logs.

### Pitfall 2: Bloqueio de rede quebra o preset colorido
**What goes wrong:** Após interceptar/abortar a rede, o "colorido" perde Mulish/JetBrains Mono (Google Fonts) e os ícones bootstrap (jsDelivr); o PDF sai com fonte serifada do sistema e quadrados no lugar de ícones.
**Why it happens:** `quote-default.hbs` linhas 34-37 referenciam CDNs externas.
**How to avoid:** Inlinar fontes via `@font-face` base64 (ou usar `fonts-liberation` já presente no Dockerfile) e trocar `<i class="bi ...">` por SVG inline/Unicode nos 3 presets. Testar o preview com rede bloqueada antes de ativar.
**Warning signs:** Preview com tipografia diferente da tela / ícones ausentes.

### Pitfall 3: Helpers globais vazando para templates de usuário
**What goes wrong:** Handlebars registra helpers no objeto global (`Handlebars.registerHelper`); um helper perigoso registrado em qualquer lugar fica disponível para o template do usuário.
**Why it happens:** A instância `Handlebars` importada é singleton de processo.
**How to avoid:** `knownHelpersOnly: true` + não registrar helpers de I/O. Idealmente usar `Handlebars.create()` para uma instância isolada por render de upload, sem helpers custom.
**Warning signs:** Template referenciando helper não documentado e funcionando.

### Pitfall 4: Prisma migration em produção / seed dos presets
**What goes wrong:** Tabela `PdfTemplate` criada mas sem os 3 presets → nenhum ativo → fallback dispara sempre (ok), mas a galeria fica vazia.
**How to avoid:** Seed idempotente dos 3 presets no `bootstrap-runtime.sh` (mesmo lugar onde rodam as migrations); marcar o "colorido" como ativo por padrão para preservar comportamento atual.

### Pitfall 5: Segredo de admin vazando para o browser
**What goes wrong:** Colocar `ADMIN_API_KEY` em `NEXT_PUBLIC_*` ou enviá-lo ao client.
**How to avoid:** A chave fica só em variável server-side do Next; o route handler a injeta via `backendFetch` (mesmo padrão de `INTERNAL_API_KEY`). O browser nunca a vê.

## Code Examples

### Resolver template ativo em renderHtml (mantendo fallback D-05)
```typescript
// Source: codebase quotes-pdf-storage.service.ts [VERIFIED: codebase]
private async resolveTemplateSource(explicit?: string): Promise<string> {
  if (explicit) return explicit;                       // preview server-side
  const active = await this.pdfTemplates.getActiveSource(); // Prisma
  if (active) return active;
  // fallback existente (D-05): EMPRESA_PDF_TEMPLATE_PATH -> quote-default.hbs -> string
  const customPath = this.configService.get<string>("EMPRESA_PDF_TEMPLATE_PATH");
  if (customPath) {
    if (!existsSync(customPath)) throw new InternalServerErrorException(`...`);
    return readFileSync(customPath, "utf-8");
  }
  const defaultPath = path.resolve(process.cwd(), "apps/backend/templates/quote-default.hbs");
  return existsSync(defaultPath) ? readFileSync(defaultPath, "utf-8") : QUOTES_PDF_HTML_TEMPLATE;
}
```
Nota: `renderHtml` passa a ser `async` (hoje é síncrono) — `generateAndStore` já é async e o chama com `await` trivial.

### Validação/sanitização de upload
```typescript
// Source: sanitize-html docs [CITED: github.com/apostrophecms/sanitize-html]
import sanitizeHtml from "sanitize-html";
function validateTemplateUpload(raw: string): string {
  if (raw.length > 512 * 1024) throw new BadRequestException("Template excede 512KB.");
  const cleaned = sanitizeHtml(raw, {
    allowedTags: false,        // permitir tags de layout, MAS:
    allowedAttributes: false,
    disallowedTagsMode: "discard",
    exclusiveFilter: () => false,
  });
  // Bloquear handlers de evento e protocolos perigosos remanescentes:
  if (/\son\w+\s*=/i.test(raw) || /javascript:/i.test(raw) || /<script/i.test(raw))
    throw new BadRequestException("Template contém script/eventos não permitidos.");
  return raw; // guardar o original (Handlebars) mas só após passar nas checagens
}
```
**Decisão de design:** sanitização agressiva pode destruir CSS/`{{...}}` legítimos. Recomenda-se: (a) rejeitar (não silenciosamente limpar) uploads com `<script>`, `on*=`, `javascript:`, `<iframe>`, `<object>`, `<embed>`, `file:`; (b) confiar no bloqueio de rede do Puppeteer + compile restrito como defesa de runtime. Confirmar abordagem com `/gsd-secure-phase`.

## Open Questions

1. **Sandbox do Puppeteer: habilitar ou aceitar `--no-sandbox` documentado?**
   - What we know: sandbox é a principal barreira; o Dockerfile roda como root com `--no-sandbox`.
   - What's unclear: o ambiente de deploy (qual host/orquestrador) suporta non-root + seccomp sem fricção.
   - Recommendation: tentar non-root + sandbox; se inviável a curto prazo, registrar a postura "rede bloqueada + sanitização + compile restrito" como compensatória em `/gsd-secure-phase`.

2. **`ADMIN_API_KEY` deve entrar em `REQUIRED_ENV_VARS`?**
   - What we know: tornar obrigatória = fail-fast; mas quebra deploys que não usam a tela de admin.
   - Recommendation: opcional; o `AdminAuthGuard` recusa se não definida (sem chave configurada = endpoints de escrita 401), preservando deploys que não usam o recurso.

3. **Sanitização: rejeitar vs limpar uploads?**
   - Recommendation: rejeitar com mensagem clara (não limpar silenciosamente) — preserva o CSS legítimo do template e evita PDFs visualmente quebrados. Confirmar em `/gsd-secure-phase`.

### Known Threat Patterns for NestJS + Handlebars + Puppeteer

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| RCE via template (prototype pollution / helperMissing) | Elevation of Privilege | Handlebars 4.7.9 + `knownHelpersOnly` + defaults de proto + `allowCallsToHelperMissing:false` |
| SSRF (metadata 169.254.169.254, IPs internos) | Information Disclosure | `setRequestInterception` abortando toda rede no render |
| Exfiltração de dados via beacon/img externo | Information Disclosure | Mesmo bloqueio de rede + sanitização de tags |
| XSS armazenado no template (`<script>`, `on*`) | Tampering | Rejeição na validação de upload |
| Leitura de arquivo local (`file://`, `@import`) | Information Disclosure | Bloqueio de `file://` na interceptação |
| Comprometimento do host via exploit do renderizador | Elevation of Privilege | Sandbox + non-root + seccomp (ou postura compensatória documentada) |
| Acesso não autorizado à tela/endpoints de admin | Spoofing | `AdminAuthGuard` + proxy server-side que esconde a chave |
| DoS por template gigante / loop de render | Denial of Service | Limite de tamanho (512KB) + `timeout` curto no Puppeteer + Throttler já global |
