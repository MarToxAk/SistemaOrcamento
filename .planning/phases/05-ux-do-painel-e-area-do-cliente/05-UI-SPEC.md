### Plan 5.1 — Internal Panel: Filters + Toast Feedback

**Status Filter Bar**
- Component: Bootstrap `nav nav-pills` with `flex-wrap gap-1 mb-3`
- Items: button elements (not `<a>`) with `nav-link` class
- Active state: `active` class on the selected filter
- Filter values: `""` (Todos), `"PENDENTE"`, `"ENVIADO"`, `"APROVADO"`, `"EM_PRODUCAO"`, `"PRONTO_PARA_ENTREGA"`, `"ENTREGUE"`, `"CANCELADO"`
- Filter labels in Portuguese: "Todos", "Pendente", "Enviado", "Aprovado", "Em Produção", "Pronto p/ Entrega", "Entregue", "Cancelado"
- Placement: above the table, inside `.orcamento-section`, before the `table-responsive` div
- Active pill color: Bootstrap default (blue `#0d6efd`) — do NOT use `#7dc8aa` for active filter; accent is reserved for CTA only
- "Todos" filter is selected by default on page load

**Toast Container (add once per page)**
```html
<div id="toast-container" class="toast-container position-fixed bottom-0 end-0 p-3" style="z-index:1100"></div>
```
Placement: inside the outermost JSX fragment, after the last `<div>`.

**Toast Specification**
- Trigger: after successful status change via `handleStatusChange`
- Type on success: `text-bg-success` (Bootstrap green)
- Type on error: `text-bg-danger` (Bootstrap red — but errors already go to `quotesError` alert; toast is success-only)
- Delay: 3500ms auto-dismiss
- Dismiss: `btn-close btn-close-white` button included in toast markup
- Guard: `typeof (window as any).bootstrap !== 'undefined'` check before instantiating
- Fallback if bootstrap not loaded: add toast markup to a React state array rendered as Bootstrap `.toast` elements (no imperative API)

**Loading State — quote list**
- While `loadingQuotes === true`: show single `<tr>` with `<td colSpan={7}>` containing a `spinner-border spinner-border-sm me-2` + "Carregando orçamentos..." text
- While `statusSavingId === quote.id`: disable the status select, show "Salvando..." as the default option — existing behavior, keep

**Data shape fix**
- Current: `setQuotes(Array.isArray(data) ? data : [])` — incorrect after Phase 3 changes `list()` response to `{ total, data: [...] }`
- Fixed: `setQuotes(Array.isArray(data) ? data : (data as any)?.data ?? [])`
- Also expose `total` in state for future pagination display

### Plan 5.2 — Form Validation: New Quote

**Validation trigger**
- Add `const [submitted, setSubmitted] = useState(false)` to the form component
- On form submit: call `setSubmitted(true)` first, then `form.checkValidity()`
- Apply `was-validated` as a React className: `className={\`needs-validation \${submitted ? "was-validated" : ""}\`}`
- Do NOT use `formRef.current.classList.add()` — React className approach is more stable across re-renders

**Required fields and their error messages**

| Field ID | Field label | Error message |
|----------|------------|---------------|
| `clienteNome` | Nome do Cliente | "Nome do cliente é obrigatório." |
| `clienteTelefone` | Telefone | "Telefone é obrigatório." |
| `vendedor` | Vendedor | "Informe o nome do vendedor." |
| `validade` | Validade da Proposta | "Informe a validade da proposta." |
| `prazoEntrega` | Prazo de Entrega | "Informe o prazo de entrega." |
| `condPagamento` | Condição de Pagamento | Not required — select always has a valid default value |

**Per-field markup pattern** (add `<div class="invalid-feedback">` immediately after each `<input required>`)
```tsx
<input type="text" className="form-control" id="clienteNome" required ... />
<div className="invalid-feedback">Nome do cliente é obrigatório.</div>
```

**Global error banner** — keep existing `<div className="alert alert-danger">` at top of form for API errors after submission. This is separate from inline field validation.

**Required field visual indicator**
- Add `*` to form labels of required fields: e.g. `<label>Nome do Cliente <span className="text-danger">*</span></label>`
- No asterisk legend needed — the error messages are self-explanatory

**Valid state** — do NOT add green borders (`is-valid` class) on success. Bootstrap `was-validated` adds green borders to valid fields by default; suppress this with CSS:
```css
.needs-validation.was-validated .form-control:valid { border-color: #ced4da; background-image: none; }
.needs-validation.was-validated .form-select:valid { border-color: #ced4da; background-image: url("data:image/svg+xml,..."); }
```
Rationale: green borders on every valid field create visual noise; only show red on invalid.

## Interactions and States

### Loading States

| Context | Trigger | Component | Copy |
|---------|---------|-----------|------|
| Quote list fetch | `loadingQuotes === true` | `spinner-border-sm` in table row | "Carregando orçamentos do contato..." |
| Status save | `statusSavingId === quote.id` | Select `disabled` + "Salvando..." option | "Salvando..." |
| PDF generation | `pdfLoadingId === quote.id` | Button `disabled` + spinner | "Gerando..." |
| Chatwoot validation | `validationState === "checking"` | Full-page `spinner-border` (existing) | "Autenticando com Chatwoot..." |
| Customer approve action | `state === "loading"` | Button `disabled` + `spinner-border-sm` | "Aprovando..." |
| Customer status page fetch | `loading === true` | Card body `spinner-border` centered | "Carregando..." |

### Error States

| Context | Component | Persistence |
|---------|-----------|-------------|
| Quote list API error | `alert-danger` at top of `.orcamento-section` | Until next action (existing behavior) |
| Status change error | Toast `text-bg-danger` (3500ms) | Auto-dismiss |
| Form validation error | Inline `invalid-feedback` per field (Bootstrap CSS) | Until field is corrected and resubmitted |
| API submission error on new quote form | `alert-danger` at top of form | Until next submit attempt |
| Customer approve error | `alert-danger` replacing card body | Permanent (no retry — user must request new link) |
| Customer status not found | `alert-danger` replacing card body | Permanent |

### Success Feedback

| Context | Component | Copy |
|---------|-----------|------|
| Status change in list page | Toast `text-bg-success` (3500ms) | "Status atualizado para [novo status label]." |
| Status change in status page | Toast `text-bg-success` (3500ms) | "Status atualizado para [novo status label]." |
| New quote created | Existing `alert-success` + redirect after 600ms | "Orçamento gerado com sucesso!" (existing) |
| Customer approval success | `alert-success` replacing card body | See copywriting section |

### Form Validation UX Flow

1. User clicks submit.
2. `setSubmitted(true)` fires — adds `was-validated` to form.
3. `form.checkValidity()` returns false → stop; Bootstrap CSS shows red borders and `invalid-feedback` text.
4. User corrects field → `:invalid` pseudo-class clears → red border disappears immediately (CSS re-evaluates on input).
5. User clicks submit again → `form.checkValidity()` returns true → proceed with API call.
6. API error → show `alert-danger` at form top. Do NOT remove `was-validated` class — keep validation visible.
7. API success → existing redirect behavior.

### Filter Interaction Flow

1. User clicks a filter pill (e.g. "Aprovado").
2. `setActiveFilter("APROVADO")` fires.
3. `useEffect` watching `[validationState, chatwootContactId, conversationId, activeFilter]` triggers.
4. Query is built: `const query = new URLSearchParams(); if (chatwootContactId) query.set("chatwootContactId", ...); if (activeFilter) query.set("status", activeFilter);`
5. `loadingQuotes` set to true → spinner appears in table.
6. Fetch completes → table re-renders with filtered rows.
7. If result is empty → empty state message appears (see copywriting).
8. "Todos" filter (`activeFilter === ""`) makes no `status` param → returns all statuses for contact.
