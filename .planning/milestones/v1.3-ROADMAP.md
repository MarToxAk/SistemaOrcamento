# Milestone v1.3: Estabilidade de Migrations no Docker Compose

**Status:** ✅ SHIPPED 2026-05-03
**Phases:** 9-10
**Total Plans:** 4

## Overview

Milestone focado em eliminar falhas de migration ao atualizar containers Docker Compose na VPS. A raiz do problema era uma race condition: o backend iniciava antes do PostgreSQL estar pronto para aceitar conexões, causando falha na execução de `prisma migrate deploy`. A solução incluiu um readiness check explícito, uma sequência de bootstrap determinista e um runbook operacional para padronizar atualizações futuras.

## Phases

**Goal:** Eliminar falhas de migration em atualizações de container garantindo execução idempotente e sincronizada com prontidão do banco.
**Depends on:** Phase 8
**Plans:** 2 plans

Plans:

- [x] 09-01-PLAN.md — Bootstrap runtime com readiness check do banco antes de migrate/start

**Requirements covered:** MIG-01, MIG-02, MIG-03

**Success criteria:**
1. Atualização via docker compose não falha com erro de migration previamente aplicada ✅

**Artifacts:**
- `apps/backend/scripts/wait-for-db.js` — polling PostgreSQL com timeout configurável
- `apps/backend/scripts/bootstrap-runtime.sh` — sequência: wait-for-db → prisma:generate → prisma:deploy → start
- `apps/backend/Dockerfile` — CMD alterado para bootstrap-runtime.sh
- `deploy/docker-compose.vps.yml` — healthcheck postgres + depends_on com service_healthy
- `deploy/stack.env.example` — WAIT_FOR_DB_TIMEOUT_MS e WAIT_FOR_DB_INTERVAL_MS

## Milestone Summary

**Key Decisions:**
- Manter `prisma migrate deploy` em produção (sem migrate reset ou force)
- Resolver no Docker Compose existente sem mudar plataforma de deploy
- Separar readiness de migration para facilitar diagnóstico isolado
- Timeout configurável por variável de ambiente (sem hardcode)

**Issues Resolved:**
- Race condition backend/postgres em `docker compose up -d` causando falha de migration
- Ausência de procedimento reproduzível de update na VPS

**Issues Deferred:**
- Validação de cold start em ambiente local (Docker CLI indisponível neste ambiente de dev)
- Smoke test automatizado no pipeline CI (requer runner com Docker)

**Tech Debt:**
- Script `verify-deploy-health.ps1` requer Docker CLI no host — não roda em runners sem Docker
- Runbook é manual; automação completa via script bash ficou fora de escopo v1.3
